# Phase 0 用户手册：项目初始化

> 本手册用于新项目启动时，按标准流程生成所有基础 context。
> 完成 Phase 0 后，gdd.md 和 tdd.md 不再有"待定"，所有 Track 的 prompt 才能正常工作。
>
> 系列手册：
> - **Phase 0: 项目初始化** ← 你在这里
> - Track A: 系统/代码
> - Track B: UI/UX（待编写）
> - Track C: 视觉美术（待编写）
> - Track D: 音频（待编写）

### 预计耗时

| 步骤 | 预计时间 | 说明 |
|------|----------|------|
| Step 1-2 | 1-2 小时 | 最核心，值得反复打磨 |
| Step 3-4 | 30-60 分钟 | 基于前两步展开 |

## Step 1：游戏概念

### 目标

确定游戏的核心身份：它是什么、给谁玩、凭什么吸引人。

### 工具

ChatGPT

### 你需要准备

- 你的灵感来源（一款游戏、一部电影、一个梦、一句话都行）
- 想要的"感觉"（紧张？治愈？爽快？）
- 1-3 个参考游戏

### Prompt

```
我要启动一个新的独立游戏项目，使用 Godot 4.3 + GDScript，目标平台 PC。
我是独立开发者，一个人做。

我的灵感来源：{{你的灵感，随便怎么描述都行}}
想要的感觉：{{玩家应该感受到什么}}
参考游戏：{{1-3 个参考游戏，说明参考了哪方面}}

请帮我收束出一个清晰的游戏概念。

## 输出格式（严格按以下结构）
- 游戏名称：（给一个暂定名，中英文各一个）
- 类型：（如 2D 平台跳跃、俯视角射击、Roguelike、解谜）
- 目标平台：PC (Windows)
- 引擎：Godot 4.3
- 核心玩法一句话描述：（不超过 20 字）
- 目标玩家：（描述目标受众，如"喜欢 Celeste 的硬核平台跳跃玩家"）
- 差异化卖点：（这个游戏和同类游戏的最大区别是什么）

## 额外要求
- 如果我的想法太模糊，帮我提炼；如果太宽泛，帮我收窄
- 名称不需要完美，暂定名就行
- 一句话描述必须让一个没玩过的人立刻理解"这个游戏是干什么的"
```

### 期望输出示例

```
- 游戏名称：暗影信使 / Shadow Courier
- 类型：2D 平台动作 + 潜行
- 目标平台：PC (Windows)
- 引擎：Godot 4.3
- 核心玩法一句话描述：在黑暗中穿越关卡，用光影操控躲避敌人
- 目标玩家：喜欢 Celeste 操作感和 Mark of the Ninja 潜行感的玩家
- 差异化卖点：光影不只是视觉效果，而是核心操控机制
```

### 与 ChatGPT 迭代

第一轮输出后，用以下问题深挖：

- "核心玩法的 30 秒体验是什么？玩家打开游戏的前 30 秒在干什么？"
- "如果只能保留一个机制，保留哪个？"
- "这个游戏最容易做失败的点在哪？"
- "MVP 最少需要什么才能让人觉得'好玩'？"

### 保存

把最终结果填入 `docs/gdd.md` 的 **"概览"** 章节，替换所有"待定"。

### 质量门

```
□ 游戏名称已确定（哪怕是暂定名）
□ 类型明确，不含歧义
□ 一句话描述不超过 20 字，外行人能看懂
□ 能回答"玩家前 30 秒在干什么"
□ 能回答"和同类游戏的区别是什么"
□ gdd.md "概览"章节已更新，无"待定"
```

## Step 2：核心机制 + 游戏循环

### 目标

把"好玩"拆解成具体的机制设计，定义玩家从打开游戏到结束的完整体验流。

### 工具

ChatGPT 讨论 → Claude 结构化

### 你需要准备

- Step 1 完成的概览章节

### Prompt（ChatGPT — 讨论阶段）

```
## 上下文
[复制 gdd.md "概览"章节]

## 任务
请帮我设计这个游戏的核心机制和游戏循环。

## 我需要你回答
### 核心机制
这个游戏有哪些核心机制？每个机制需要说明：
- 描述：这个机制是什么
- 玩家输入：玩家做什么操作
- 系统响应：游戏如何响应
- 反馈循环：玩家怎么知道自己做对了/做错了

### 游戏循环
分三个层次描述：
1. **30 秒体验**：玩家前 30 秒在干什么（moment-to-moment gameplay）
2. **核心循环**：单局/单关的完整流程（开始 → 过程 → 结束 → 奖励）
3. **长期循环**：跨局/跨关的成长和进度（什么驱动玩家继续玩）

## 要求
- 核心机制不超过 3 个（独立开发者资源有限）
- 明确哪个是主机制，哪些是辅机制
- 30 秒体验必须具体到"按什么键 → 看到什么 → 感受到什么"
```

### 与 ChatGPT 迭代

关键追问：

- "这些机制之间怎么互动？有没有协同效果？"
- "核心循环中玩家如果失败了，惩罚是什么？会不会太重/太轻？"
- "长期循环有没有尽头？玩 10 小时后还有什么新东西？"
- "哪个机制最难做？能不能在 MVP 里先去掉？"

### Prompt（Claude — 收束阶段）

讨论收敛后，把结论交给 Claude 结构化：

```
## 上下文
[复制 gdd.md "概览"章节]

## 讨论结论
以下是我和 ChatGPT 讨论确定的核心机制和游戏循环设计：
{{粘贴 ChatGPT 讨论的最终结论}}

## 任务
请按以下格式结构化输出，我会直接填入 gdd.md。
如果有模糊或矛盾的地方，指出来让我补充。

## 输出格式
### 核心机制

#### 机制 1：[名称]（主机制）
- 描述：
- 玩家输入：
- 系统响应：
- 反馈循环：

#### 机制 2：[名称]（辅机制）
- 描述：
- 玩家输入：
- 系统响应：
- 反馈循环：

### 游戏循环

#### 30 秒体验
[具体到按键和画面描述]

#### 核心循环
[单局/单关的完整流程]

#### 长期循环
[跨局/跨关的成长系统]
```

### 保存

把 Claude 的输出填入 `docs/gdd.md` 的 **"核心机制"** 和 **"游戏循环"** 章节。

### 质量门

```
□ 核心机制不超过 3 个
□ 每个机制都有完整的：描述 / 玩家输入 / 系统响应 / 反馈循环
□ 明确标注了主机制和辅机制
□ 30 秒体验具体到按键级别
□ 核心循环有完整的闭环（入口 → 过程 → 出口）
□ 长期循环有明确的驱动力
□ gdd.md 对应章节已更新，无"待定"
```

## Step 3：内容设计

### 目标

定义游戏中的角色/实体和关卡/世界结构。

### 工具

ChatGPT

### 你需要准备

- gdd.md 的"概览"和"核心机制"章节（Step 1-2 产出）

### Prompt

```
## 上下文
[复制 gdd.md "概览"章节]
[复制 gdd.md "核心机制"章节]
[复制 gdd.md "游戏循环"章节]

## 任务
请帮我设计这个游戏的角色/实体和关卡/世界结构。

### 角色 / 实体
列出游戏中所有主要的角色和实体。每个需要说明：
- 名称
- 角色类型：（玩家角色 / 敌人 / NPC / 环境实体 / 可交互物品）
- 核心行为：（它能做什么 / 会做什么）
- 与核心机制的关系：（它怎么参与核心玩法）
- MVP 优先级：（P0 必须有 / P1 应该有 / P2 可以没有）

### 关卡 / 世界设计
- 关卡结构：（线性关卡 / 开放世界 / 随机生成 / hub world）
- 难度曲线：（如何渐进）
- 关卡数量预估（MVP）：
- 每关时长预估：
- 关卡间的进度系统：（什么解锁下一关）

## 要求
- 角色/实体数量要控制在独立开发者可承受范围
- 明确标注 MVP 必须有的和可以后加的
- 关卡设计要和核心循环匹配
```

### 保存

填入 `docs/gdd.md` 的 **"角色 / 实体"** 和 **"关卡 / 世界设计"** 章节。

### 质量门

```
□ 角色/实体列表完整，有优先级标注
□ 每个角色/实体都和核心机制有关联
□ P0 角色/实体数量可控（MVP 不超过 5-8 种）
□ 关卡结构明确
□ 难度曲线有描述
□ gdd.md 对应章节已更新，无"待定"
```

## Step 4：UI/UX 规划

### 目标

定义游戏需要哪些 UI 界面、每个界面的用途和优先级。

### 工具

ChatGPT

### 你需要准备

- gdd.md 的概览 + 核心机制 + 内容设计章节（Step 1-3 产出）

### Prompt

```
## 上下文
[复制 gdd.md "概览"章节]
[复制 gdd.md "核心机制"章节]
[复制 gdd.md "角色 / 实体"章节]

## 任务
请帮我规划这个游戏需要的所有 UI 界面。

## 对于每个界面，请说明：
- 界面名称
- 用途：一句话说明
- 进入方式：玩家怎么打开这个界面
- 核心信息：这个界面需要展示什么
- 核心操作：玩家在这里能做什么
- 优先级：P0（MVP 必须）/ P1（完整体验需要）/ P2（锦上添花）

## 额外要求
- HUD 需要特别说明哪些信息常驻显示
- 考虑从主菜单到游戏内再到退出的完整 UI 流程
- 不要设计过多界面，独立开发者每个界面都是成本

## 输出格式
### UI 界面清单
| 界面 | 用途 | 进入方式 | 优先级 |
|------|------|----------|--------|

### HUD 详细设计
- 常驻信息：[列出]
- 条件显示信息：[列出，以及显示条件]

### UI 流程
```
主菜单 → 开始游戏 → 游戏内(HUD) → ESC → 暂停菜单 → ...
```
```

### 保存

填入 `docs/gdd.md` 的 **"UI/UX"** 章节，更新界面清单表格。

### 质量门

```
□ 界面清单完整，覆盖从启动到退出的全流程
□ 每个界面有明确优先级
□ HUD 信息层级清晰（常驻 vs 条件显示）
□ P0 界面不超过 4-5 个
□ gdd.md "UI/UX"章节已更新，无"待定"
```

## Step 5：美术方向

### 目标

确定视觉风格方向，建立 AI 生图的基底 prompt，让所有后续美术资产风格一致。

### 工具

ChatGPT

### 你需要准备

- gdd.md 的概览章节
- 你的风格偏好（像素风？手绘？低多边形？）
- 1-3 个视觉参考（游戏截图、插画、电影画面都行）

### Prompt

```
## 上下文
[复制 gdd.md "概览"章节]

## 任务
我需要为这个游戏确定完整的美术风格规格。
美术资产主要通过 AI 生成（ComfyUI / Stable Diffusion / Midjourney），
所以需要风格可被 AI 稳定复现。

我的偏好：
- 风格方向：{{如：像素风 / 手绘 / 低多边形 / 卡通渲染}}
- 情绪基调：{{如：明快欢快 / 阴暗压抑 / 温馨治愈 / 硬核冷峻}}
- 视觉参考：{{列出 1-3 个参考游戏或作品，说明参考了哪方面}}

## 输出格式（严格按此结构，我会直接填入 gdd.md）

### 风格方向
- 主风格：（如 pixel art, 16-bit style）
- 子风格参考：（如 Celeste 的干净像素 + Dead Cells 的流畅动画）

### 技术参数
- 游戏分辨率：（如 320x180 → 缩放至 1920x1080）
- 角色精灵尺寸：（如 32x32 / 64x64）
- 动画帧率：（如 12fps 像素动画）
- 缩放模式：（如 nearest neighbor / linear）

### 调色板
- 主色：（hex 值 + 用途，如 #2B2D42 深蓝灰 — 背景/阴影）
- 辅色：（hex 值 + 用途）
- 强调色：（hex 值 + 用途，如 #EF233C 亮红 — 危险/伤害反馈）
- 参考色板：（如 ENDESGA-32, Pico-8, 自定义）

### 角色设计
- 风格描述：（如"简约像素，大头小身，4 色以内用色"）
- 尺寸规范：（精灵尺寸、动画帧数基准）

### 环境设计
- 风格描述：（如"分层视差背景，主体用 Tilemap"）
- Tile 尺寸：（如 16x16）

### UI 风格
- 风格描述：（如"扁平半透明面板，像素字体"）
- 字体建议：（如 pixel font, 8px / 16px）

### AI 生图基底 Prompt
正向 prompt 模板：
```
[风格标签], [主体描述], [配色参考], [情绪标签], [质量标签]
```
示例：
```
pixel art, 16-bit style, [subject], [action], limited color palette, clean lines, no anti-aliasing, game asset, transparent background
```

负向 prompt 模板：
```
[需要排除的标签]
```
示例：
```
blurry, realistic, 3d render, watermark, text, signature, low quality, deformed, extra limbs
```

### 风格一致性检查清单
- [ ] 所有角色使用相同精灵尺寸
- [ ] 颜色在调色板范围内
- [ ] 光影方向统一（标注方向，如左上 45°）
- [ ] 描边风格统一（有/无、粗细、颜色）
```

### 与 ChatGPT 迭代

关键追问：

- "调色板的对比度够不够？低视力玩家能分清关键元素吗？"
- "这套风格能被 Stable Diffusion 稳定复现吗？需要用什么 checkpoint / LoRA？"
- "如果用这套正向 prompt 生成 5 张图，风格一致性能有多高？"

### 保存

把输出完整填入 `docs/gdd.md` 的 **"美术风格"** 章节（覆盖所有子章节）。

### 质量门

```
□ 主风格和子风格都有具体描述（不只是"像素风"这三个字）
□ 技术参数全部确定（分辨率、精灵尺寸、帧率、缩放模式）
□ 调色板有具体 hex 值，不少于 3 色
□ AI 生图正向和负向 prompt 都已填写
□ 风格一致性检查清单已完善
□ gdd.md "美术风格"全部子章节已更新，无"待定"
```

## Step 7：里程碑定义

### 目标

把前面所有设计决策收束为明确的里程碑范围，让你知道 MVP 做什么、不做什么。

### 工具

## Step 9：一致性校验

### 目标

交叉检查所有文档，确保没有矛盾、遗漏或不一致。

### 工具

Claude Code

### Prompt

```
请阅读以下所有文件，做一致性交叉检查：
- docs/gdd.md
- docs/tdd.md
- ai/context/architecture.md
- CLAUDE.md

## 检查维度

### 1. 完整性检查
- gdd.md 是否还有"待定"字样？
- tdd.md 是否还有"待定"字样？
- gdd.md 中的每个功能是否都在 tdd.md 中有对应的系统？
- tdd.md 中的每个系统是否都能对应到 gdd.md 中的功能？

### 2. 一致性检查
- gdd.md 中的角色/实体列表 和 tdd.md 中的系统列表是否对得上？
- gdd.md 中的 UI 界面清单 和 tdd.md 中是否有对应的 UI 系统？
- 美术参数（分辨率、精灵尺寸）是否在 tdd.md 的性能考量中被考虑？
- CLAUDE.md 中的项目结构描述是否和实际一致？

### 3. 可行性检查
- MVP 范围内的功能，对应的系统数量是否合理（独立开发者 2-4 周）？
- 是否有系统过于复杂，应该拆分或推迟到 Alpha？

## 输出格式
### 问题清单
| 级别 | 文件 | 问题描述 | 建议修复 |
|------|------|----------|----------|
| 🔴 矛盾 | | | |
| 🟡 遗漏 | | | |
| 🟢 建议 | | | |

### 遗留的"待定"项
（列出所有仍然包含"待定"的位置）

### 结论
Phase 0 是否可以通过？如果不能，列出必须先修复的问题。
```
