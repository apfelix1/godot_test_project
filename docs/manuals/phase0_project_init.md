# Phase 0 用户手册：项目初始化

> 本手册用于新项目启动时，按标准流程生成所有基础 context。
> Phase 0 分为两个阶段：
> - **Phase 0a 完成后**，gdd.md 核心章节和 tdd.md 核心系统就绪，Track A 可正式执行，Track B（UI 预研）和 Track C 的 C1（风格预研）可开始。
> - **Phase 0b 完成后**，所有文档无"待定"，Track B/C/D 可正式执行。
>
> 系列手册：
> - **Phase 0: 项目初始化** ← 你在这里
> - Track A: 系统/代码
> - Track B: UI/UX（Phase 0a 后可预研，手册待编写）
> - Track C: 视觉美术（C1 风格预研可在 0a 后开始，手册待编写）
> - Track D: 音频（需 Phase 0b，手册待编写）

## 共享协作文件

在整个 Phase 0 过程中，默认使用以下文件协作：

- `AGENTS.md`：仓库级规则与接手顺序
- `docs/project_status.md`：当前真实状态
- `docs/tasks.md`：任务面板与状态机
- `docs/ai_handoff.md`：AI 交接记录
- `CLAUDE.md`：Claude 兼容入口（若使用 Claude 系工具）

## 流程总览

### Phase 0a（原型初始化）

```
Step 1 → Step 2 → Step 3(轻量) → Step 4(最小) → Step 7(仅MVP) → Step 8(核心) → Step 9(原型)
→ ✅ 0a 完成 → Track A 开始 → 第一个 P0 功能完成后 → 灰盒试玩
```

允许"待定"：美术风格、音频方向、非 MVP 内容、非 MVP 系统。

```
Step 3(完整) → Step 4(完整) → Step 5 → Step 6 → Step 7(完整) → Step 8(完整) → Step 9(全量)
→ ✅ 0b 完成 → Track B/C/D 可正式执行
```

目标：填充所有生产细节，消除全部"待定"。
要求：gdd.md 和 tdd.md 中零"待定"。

## Step 1：游戏概念

### 目标

确定游戏的核心身份：它是什么、给谁玩、凭什么吸引人。

### 工具

ChatGPT

### 你需要准备

- 你的灵感来源（一款游戏、一部电影、一个梦、一句话都行）
- 想要的"感觉"（紧张？治愈？爽快？）
- 1-3 个参考游戏

### Prompt

```
我要启动一个新的独立游戏项目，使用 Godot 4.3 + GDScript，目标平台 PC。
我是独立开发者，一个人做。

我的灵感来源：{{你的灵感，随便怎么描述都行}}
想要的感觉：{{玩家应该感受到什么}}
参考游戏：{{1-3 个参考游戏，说明参考了哪方面}}

请先与我讨论进行brainstorm，最后帮我收束出一个清晰的游戏概念。

## 输出格式（严格按以下结构）
- 游戏名称：（给一个暂定名，中英文各一个）
- 类型：（如 2D 平台跳跃、俯视角射击、Roguelike、解谜）
- 目标平台：PC (Windows)
- 引擎：Godot 4.3
- 核心玩法一句话描述：（不超过 20 字）
- 目标玩家：（描述目标受众，如"喜欢 Celeste 的硬核平台跳跃玩家"）
- 差异化卖点：（这个游戏和同类游戏的最大区别是什么）

## 额外要求
- 如果我的想法太模糊，帮我提炼；如果太宽泛，帮我收窄
- 名称不需要完美，暂定名就行
- 一句话描述必须让一个没玩过的人立刻理解"这个游戏是干什么的"
```

### 期望输出示例

```
- 游戏名称：暗影信使 / Shadow Courier
- 类型：2D 平台动作 + 潜行
- 目标平台：PC (Windows)
- 引擎：Godot 4.3
- 核心玩法一句话描述：在黑暗中穿越关卡，用光影操控躲避敌人
- 目标玩家：喜欢 Celeste 操作感和 Mark of the Ninja 潜行感的玩家
- 差异化卖点：光影不只是视觉效果，而是核心操控机制
```

### 与 ChatGPT 迭代

第一轮输出后，用以下问题深挖：

- "核心玩法的 30 秒体验是什么？玩家打开游戏的前 30 秒在干什么？"
- "如果只能保留一个机制，保留哪个？"
- "这个游戏最容易做失败的点在哪？"
- "MVP 最少需要什么才能让人觉得'好玩'？"

### 保存

把最终结果填入 `docs/gdd.md` 的 **"概览"** 章节，替换所有"待定"。

### 质量门

```
□ 游戏名称已确定（哪怕是暂定名）
□ 类型明确，不含歧义
□ 一句话描述不超过 20 字，外行人能看懂
□ 能回答"玩家前 30 秒在干什么"
□ 能回答"和同类游戏的区别是什么"
□ gdd.md "概览"章节已更新，无"待定"
```

## Step 2：核心机制 + 游戏循环

### 目标

把"好玩"拆解成具体的机制设计，定义玩家从打开游戏到结束的完整体验流。

### 工具

ChatGPT 讨论 → Claude 结构化

### 你需要准备

- Step 1 完成的概览章节

### Prompt（ChatGPT — 讨论阶段）

```
## 上下文
[复制 gdd.md "概览"章节]

## 任务
请帮我设计这个游戏的核心机制和游戏循环。

## 我需要你回答
### 核心机制
这个游戏有哪些核心机制？每个机制需要说明：
- 描述：这个机制是什么
- 玩家输入：玩家做什么操作
- 系统响应：游戏如何响应
- 反馈循环：玩家怎么知道自己做对了/做错了

### 游戏循环
分三个层次描述：
1. **30 秒体验**：玩家前 30 秒在干什么（moment-to-moment gameplay）
2. **核心循环**：单局/单关的完整流程（开始 → 过程 → 结束 → 奖励）
3. **长期循环**：跨局/跨关的成长和进度（什么驱动玩家继续玩）

## 要求
- 核心机制不超过 3 个（独立开发者资源有限）
- 明确哪个是主机制，哪些是辅机制
- 30 秒体验必须具体到"按什么键 → 看到什么 → 感受到什么"
```

### 与 ChatGPT 迭代

关键追问：

- "这些机制之间怎么互动？有没有协同效果？"
- "核心循环中玩家如果失败了，惩罚是什么？会不会太重/太轻？"
- "长期循环有没有尽头？玩 10 小时后还有什么新东西？"
- "哪个机制最难做？能不能在 MVP 里先去掉？"

### Prompt（Claude — 收束阶段）

讨论收敛后，把结论交给 Claude 结构化：

```
## 上下文
[复制 gdd.md "概览"章节]

## 讨论结论
以下是我和 ChatGPT 讨论确定的核心机制和游戏循环设计：
{{粘贴 ChatGPT 讨论的最终结论}}

## 任务
请按以下格式结构化输出，我会直接填入 gdd.md。
如果有模糊或矛盾的地方，指出来让我补充。

## 输出格式
### 核心机制

#### 机制 1：[名称]（主机制）
- 描述：
- 玩家输入：
- 系统响应：
- 反馈循环：

#### 机制 2：[名称]（辅机制）
- 描述：
- 玩家输入：
- 系统响应：
- 反馈循环：

### 游戏循环

#### 30 秒体验
[具体到按键和画面描述]

#### 核心循环
[单局/单关的完整流程]

#### 长期循环
[跨局/跨关的成长系统]
```

### 保存

把 Claude 的输出填入 `docs/gdd.md` 的 **"核心机制"** 和 **"游戏循环"** 章节。

### 质量门

```
□ 核心机制不超过 3 个
□ 每个机制都有完整的：描述 / 玩家输入 / 系统响应 / 反馈循环
□ 明确标注了主机制和辅机制
□ 30 秒体验具体到按键级别
□ 核心循环有完整的闭环（入口 → 过程 → 出口）
□ 长期循环有明确的驱动力
□ gdd.md 对应章节已更新，无"待定"
```

## Step 5：美术方向

> **Phase 0b 步骤**：Phase 0a 阶段跳过此步骤，使用占位美术即可。
> 进入 Track C（视觉美术）前必须完成。

### 目标

确定视觉风格方向，建立 AI 生图的基底 prompt，让所有后续美术资产风格一致。

### 工具

ChatGPT

### 你需要准备

- gdd.md 的概览章节
- 你的风格偏好（低多边形？卡通渲染？手绘贴图？）
- 1-3 个视觉参考（游戏截图、插画、电影画面都行）

### Prompt

```
## 上下文
[复制 gdd.md "概览"章节]

## 任务
我需要为这个游戏确定完整的美术风格规格。
美术资产主要通过 AI 生成（ComfyUI / Stable Diffusion / Midjourney），
所以需要风格可被 AI 稳定复现。

我的偏好：
- 风格方向：{{如：低多边形 / 手绘贴图 / 卡通渲染 / stylized 3D}}
- 情绪基调：{{如：明快欢快 / 阴暗压抑 / 温馨治愈 / 硬核冷峻}}
- 视觉参考：{{列出 1-3 个参考游戏或作品，说明参考了哪方面}}

## 输出格式（严格按此结构，我会直接填入 gdd.md）

- 游戏分辨率：（如 1920x1080）
- 角色模型规格：（如角色高度、面数预算、单位比例）
- 动画帧率：（如 30fps / 60fps）
- 渲染与过滤设置：（如 texture filter / anti-aliasing / shadow quality）

### 调色板
```

| 1 个最爽的时刻 | 什么时候觉得"这个有意思"？ |
| 核心循环是否成立 | 30 秒体验是否和 gdd.md 描述一致？ |

2. 把结果写入 `docs/tasks.md` 的「试玩反馈」区域
4. 根据反馈决定：
   - 核心循环成立 → 继续从 MVP 范围取下一个功能
   - 核心循环不成立 → 回到 Step 2 重新设计机制，再重做第一个功能

### 质量门

### 第一个功能怎么开始？（Phase 0a 完成后即可）

Phase 0a 产出了 gdd.md 和 tdd.md 的核心章节，但此时还没有任何可运行的代码。
按以下顺序启动第一个功能：

```
1. 初始化测试脚手架（一次性）：
   - 创建 tests/ 目录
   - 创建 tests/smoke_test.gd（最小 headless 启动检查脚本）
   - 约定：godot --headless --script res://tests/smoke_test.gd 退出码 0 = 通过
2. 打开 gdd.md "里程碑" → MVP 章节
3. 从 MVP 范围中选一个最核心的系统（通常是玩家控制/移动）
4. 走迭代手册路径 A → docs/manuals/iteration_add_feature.md
   - 在 docs/tasks.md 中先创建 `draft` 状态的 Sprint
   - 真正开始编码后再把 Sprint 改成 `in_progress`
   - A-1 可以简化（MVP 里已经定义过需求了，直接用 gdd.md 的描述）
   - A-2 正常走（需要 Claude 出工程规格）
   - A-3 ~ A-6 正常走
5. Path A 完成后 → 做灰盒试玩复盘（见上方章节）
6. 核心循环验证通过后 → 走 Path C 并入主线
7. 继续从 MVP 范围取下一个功能
```

> 建议第一个功能选"能在屏幕上看到角色并操控"的最小闭环，
> 这样后续所有功能都有一个可运行的基础。

### 日常开发路由

```
需要做新功能     → 迭代手册路径 A（docs/manuals/iteration_add_feature.md）
需要加素材      → 迭代手册路径 B
修 bug          → 迭代手册路径 D
功能做完并入    → 迭代手册路径 C
需要做 UI       → Track B 用户手册（Phase 0a 后可预研，手册待编写）
需要出美术资产   → Track C 用户手册（C1 风格预研可在 0a 后开始，手册待编写）
需要做音频      → Track D 用户手册（需 Phase 0b，手册待编写）
```
