# 迭代手册：新增功能 / 素材 / 并入

> 本手册用于项目开发过程中，新增系统、素材或将完成的功能并入主线。
> Phase 0 完成后，所有迭代开发都走本手册的流程。
>
> 系列手册：
> - Phase 0: 项目初始化
> - Track A: 系统/代码
> - **迭代手册：新增功能/素材/并入** ← 你在这里
> - Track B: UI/UX（Phase 0a 后可预研，手册待编写）
> - Track C: 视觉美术（C1 风格预研可在 0a 后开始，手册待编写）
> - Track D: 音频（需 Phase 0b，手册待编写）

---

> **MVP 简化**：MVP 阶段推荐双模型工作流 — ChatGPT（发散讨论）+ Claude Code（收束、实现、review，直接读项目文件）。
> Kimi 为可选，适合后期批量模板和配置生成。

## 共享协作文件

进入迭代流程前，默认先看：

- `AGENTS.md`
- `docs/project_status.md`
- `docs/tasks.md`
- `docs/ai_handoff.md`
- `CLAUDE.md`（若使用 Claude 系工具）

## Sprint 状态机

迭代手册统一使用以下 Sprint 状态：

- `draft`
- `in_progress`
- `code_complete`
- `merged`

## 路径选择

```
我要做什么？
    │
    ├── 加新功能 / 新系统 / 新机制？
    │   └── 路径 A：新增系统（重流程）
    │
    ├── 只是加资源文件（图片/音频/数据表）？
    │   └── 路径 B：新增素材（轻流程）
    │
    ├── 修 bug？
    │   └── 路径 D：Bug 修复（快速流程）
    │
    └── 功能开发完了，要合进主线？
        └── 路径 C：并入主线
```

### 与 Track A 手册的关系

> 本手册定义"做什么"和"怎么管"（需求 → 设计 → 文档 → 拆任务 → 验证 → 并入）。
> Track A 手册定义"怎么写代码"（具体的编码 prompt 和 review 流程）。
>
> 简单说：迭代手册是项目管理层，Track A 是执行层。
> 路径 A 的 A-4（实现）直接跳转到 Track A 的 A4/A5 执行。

### 判断规则

| 信号 | 走路径 A | 走路径 B |
|------|----------|----------|
| 需要新 .gd 脚本 | ✅ | |
| 需要新信号或 EventBus 事件 | ✅ | |
| 需要修改现有系统 | ✅ | |
| 需要新 Autoload | ✅ | |
| 只是放文件到 assets/ | | ✅ |
| 替换已有素材（同名同规格） | | ✅ |
| 只改静态配置（UI 文案、颜色值） | | ✅ |
| 改行为型数据（敌人数值、掉落表、AI 参数） | | ✅ + 回归验证 |

> **行为型数据**（敌人数值、掉落表、AI 参数等）虽然走 Path B 流程，
> 但因为会影响玩法体验，B-4 验证时必须做一次回归测试（至少 F5 跑一遍相关场景）。
> 如果改动范围大到需要新增/修改脚本逻辑，应该走 Path A。

> 如果一个需求同时涉及两者（如"加一个新敌人"= 新 AI 脚本 + 新 sprite），
> 先走路径 A 把系统骨架搭好，再走路径 B 填素材。

## 路径 A：新增系统（重流程）

适用于：新功能、新系统、新机制、需要写代码的改动。

### 总览

```
A-1 需求定义     A-2 设计收束(含设计冻结)     A-3 任务拆分     A-4 实现     A-5 集成验证     A-6 文档同步(仅偏差)
ChatGPT          Claude                       Claude Code      Track A      你 + CC          你 + CC
"做什么？"   →  "怎么做？冻结设计"        →  "拆成任务"   →  "写代码"  →  "能跑吗？"   →  "偏差才更新"
```

> 统一规则：设计冻结时更新 gdd.md / tdd.md；开发进行中持续维护 tasks.md；实现偏差或并入时同步状态和相关文档。

### 前置条件

```
□ Phase 0a 已完成（gdd.md 核心章节 + tdd.md 核心系统就绪）
□ 主功能线（Path A）当前没有未完成的流程
```

### 三车道并行规则

| 车道 | 路径 | 并行规则 |
|------|------|----------|
| 主功能线 | Path A | 同时只做一个 |
| 素材/调参线 | Path B | 可与主功能线并行 |
| 紧急修复线 | Path D | 可随时插入，优先处理 |
| 合流 | Path C | 串行，一次合一个 |

> 不再要求"一次只做一个功能"。素材导入和紧急修复不需要等主功能线完成。

### A-1：需求定义

**目标**：把模糊想法变成一段清晰的功能描述。

**工具**：ChatGPT

**Prompt**：

```
## 上下文
[复制 docs/gdd.md "概览"章节]

## 已有系统
[复制 docs/tdd.md "核心系统"章节的系统名称列表]

## 需求
我想给游戏加一个新功能：{{功能名称}}
玩家体验目标：{{一句话，玩家应该感受到什么}}
触发场景：{{玩家在什么情况下会用到这个功能}}

请帮我把这个想法收束成一段清晰的功能描述。

## 输出格式
- 功能名称：
- 一句话描述：（不超过 20 字，外行人能看懂）
- 玩家体验目标：
- 核心交互：玩家做什么 → 系统怎么响应 → 什么反馈
- 与已有系统的关系：（依赖哪些、影响哪些）
- MVP 范围：（最小可用版本只需要什么）
- 明确不做：（这个功能不包含什么，防止 scope creep）
```

**与 ChatGPT 迭代**：

- "这个功能的 MVP 能再砍吗？最少需要什么？"
- "和已有的 XX 系统会冲突吗？"
- "如果不做这个功能，游戏缺什么？"（验证必要性）

**产出指标**：

| 指标 | 要求 |
|------|------|
| 一句话描述 | ≤ 20 字，无歧义 |
| 核心交互 | 完整的 输入→响应→反馈 链 |
| MVP 范围 | 有明确边界，列出"不做"项 |
| 与已有系统关系 | 列出所有依赖和影响 |

**质量门**：

```
□ 能用一句话向非开发者解释这个功能
□ 核心交互链完整（输入→响应→反馈）
□ MVP 范围有明确的"做"和"不做"
□ 已评估与现有系统的关系
```

### A-2：设计收束

**目标**：把功能描述转成工程可执行的系统规格。

**工具**：Claude（网页版或 Claude Code）

**Prompt**：

```
## 上下文
[复制 docs/tdd.md "架构概览"章节]
[复制 ai/context/architecture.md 全文]

## 编码规范
[复制 ai/context/coding_conventions.md 的"命名规则"和"信号使用原则"]

## 功能描述
{{粘贴 A-1 的完整输出}}

## 任务
请把这个功能收束为工程规格。

## 输出格式
### 系统概述
- 名称：
- 一句话职责：
- 所属目录：source/features/xxx/

### 场景 & 节点树
```
[节点树结构，不超过 3 层]
```

### 信号设计
| 信号名 | 参数 | 触发时机 | 谁监听 |
|--------|------|----------|--------|

### 数据结构
| 数据 | 类型 | 说明 |
|------|------|------|

### 与已有系统的集成
- 新增 EventBus 事件：[列出]
- 依赖的系统：[列出]
- 被依赖：[列出]
- 需要修改的现有文件：[列出文件和修改内容]

### 风险点
- [列出实现中可能踩坑的地方]
```

**审查要点**：

| 检查项 | 标准 |
|--------|------|
| 节点树深度 | ≤ 3 层，扁平优先 |
| 信号命名 | past_tense（如 health_changed） |
| 循环依赖 | 不允许，A→B→A 不可接受 |
| EventBus 使用 | 仅跨系统广播型通信，详见下方决策树 |
| 修改范围 | 列出所有需要改的现有文件 |

**质量门**：

```
□ 节点树 ≤ 3 层
□ 每个信号都有明确的触发时机和监听者
□ 无循环依赖
□ EventBus 事件仅用于跨系统
□ 所有需要修改的现有文件已列出
□ 风险点已识别
```

通信方式选择（详见 ai/context/coding_conventions.md）：
- 广播型事件（一对多）→ EventBus
- 命令式调用（一对一，目标明确）→ 直接引用
- 父子节点 → 信号向上，方法向下

**设计冻结交付物**：

设计收束完成后，将设计决策写入项目文档，保持 source of truth 一致：

| 文件 | 更新内容 | 验证方式 |
|------|----------|----------|
| docs/gdd.md | 新功能描述（来自 A-1） | 搜索功能名称能找到 |
| docs/tdd.md | 新系统规格摘要（来自 A-2） | "核心系统"章节有新条目 |
| ai/context/architecture.md | 新 EventBus 事件（如有） | 事件列表包含新事件 |

```
□ gdd.md 已更新
□ tdd.md 已更新
□ architecture.md 已更新（如有新 EventBus 事件）
□ 三个文件之间无矛盾
```

### A-3：任务拆分

**目标**：把工程规格拆成可执行的开发任务。

**工具**：Claude Code

**Prompt**：

```
请阅读：
- docs/gdd.md 中关于 {{功能名称}} 的描述
- docs/tdd.md 中关于 {{系统名称}} 的规格
- AGENTS.md 的协作规则
- ai/context/coding_conventions.md 的编码规范

请把这个功能拆成开发任务，写入 docs/tasks.md。

## 拆分规则
- 每个任务 1-3 小时可完成
- 标明涉及的文件路径
- 标明任务间依赖关系
- 按 P0/P1/P2 优先级排列
- P0 完成后必须能在 Godot 里看到基本效果

## 输出格式（追加到 docs/tasks.md）
## Sprint: {{功能名称}}
状态：draft
任务类型：code
建议执行者：Codex / Claude Code
目标：{{一句话}}

### P0 — 核心
- [ ] 任务描述 | 文件: source/xxx/ | 依赖: 无
- [ ] 任务描述 | 文件: source/xxx/ | 依赖: 任务1

### P1 — 完整
- [ ] 任务描述 | 文件: source/xxx/

### P2 — 打磨
- [ ] 任务描述
```

**产出指标**：

| 指标 | 要求 |
|------|------|
| 任务粒度 | 每个 1-3 小时 |
| 任务数量 | P0 不超过 5 个，总计不超过 12 个 |
| 具体程度 | 每个任务包含具体文件路径 |
| 依赖关系 | 无互相依赖，DAG 结构 |
| P0 可验证 | P0 全部完成后能 F5 看到效果 |

**质量门**：

```
□ 每个任务有具体的完成描述和文件路径
□ P0 任务完成后能在 Godot 里看到基本效果
□ 没有互相依赖的任务
□ tasks.md 已更新
```

### A-4：实现

**目标**：按任务单写代码。

**工具**：走 Track A 标准流程（A4-A5）

直接按 [Track A 用户手册](track_a_system_code.md) 的 A4（代码实现）和 A5（Code Review）执行。

> **迭代模式注意**：走迭代流程时，A5 只做 review 和 Godot 验证，不做 git commit。提交统一在 Path C 处理。

**关键规则**：

- 按 P0 → P1 → P2 顺序实现
- 开始第一个实际编码任务时，把 Sprint 状态改成 `in_progress`
- 每完成一个任务立刻 F5/F6 验证
- 每个任务完成后在 tasks.md 标记 `[x]`
- 核心逻辑用 Claude Code，批量数据用 Kimi

### A-5：集成验证

**目标**：确认新系统与现有系统正确集成，没有破坏已有功能。

**工具**：你在 Godot 中操作 + Claude Code 辅助

**检查清单**：

```
## 信号连接验证
□ 新增的 EventBus 事件能正确触发
□ 新增的 EventBus 事件有正确的监听者
□ 没有孤立信号（发了没人收）
□ 没有悬空连接（连了但源节点不存在）

## Autoload 验证（如新增了 Autoload）
□ project.godot 中已注册
□ 加载顺序正确（依赖的 Autoload 先加载）
□ 其他系统能正确访问

## 回归验证
□ F5 运行主场景 → 无报错
□ 已有功能逐个测试 → 没有被破坏
□ 新功能核心路径走通（从头到尾操作一遍）
□ 边界情况测试（空值、极端值、快速重复操作）

## 性能验证
□ _process 中没有重计算
□ 没有每帧创建新对象
□ 信号没有触发死循环
```

**产出指标**：

| 指标 | 要求 |
|------|------|
| 报错数 | 0（F5 运行无报错） |
| 已有功能回归 | 全部通过 |
| 新功能核心路径 | 可走通 |
| 性能 | 无明显卡顿 |

**质量门**：

```
□ 所有信号连接正确
□ F5 无报错
□ 已有功能未被破坏
□ 新功能核心路径可走通
```

### 灰盒试玩复盘（可选，推荐）

功能集成后，花 15-30 分钟试玩，记录：

| 维度 | 记录 |
|------|------|
| 新功能是否符合预期 | 和 A-2 设计的体验一致吗？ |
| 是否引入了新的困惑点 | 玩家会不会不知道该干什么？ |
| 是否影响了已有体验 | 旧功能的手感变了吗？ |

结果写入 tasks.md。如果发现根本性问题，走路径 D 修复。

### A-6：文档同步（仅偏差）

**目标**：仅当实现与 A-2 设计有偏差时，同步更新文档。如果完全按设计实现，跳过此步。

**工具**：Claude Code

**Prompt**：

无论是否有偏差，都更新：

## 要求
- 以实际代码为准更新文档
- 只更新确实有偏差的文件，不做无意义的同步
```

**产出指标**：

| 指标 | 要求 |
|------|------|
| 偏差已识别 | 列出所有实现与设计的差异 |
| 文档与代码一致 | 偏差部分已同步 |
| tasks.md | Sprint 标记为 `code_complete` |

**质量门**：

```
□ 实现偏差已识别并记录
□ 偏差部分的文档已更新
□ tasks.md 已更新
□ 改动已保存（git commit 在 Path C 统一处理）
```

## 路径 B：新增素材（轻流程）

适用于：3D 模型、贴图、材质、音频、UI 资源、数据表等资源类改动。

### 总览

```
B-1 确认规范     B-2 准备素材     B-3 导入项目     B-4 验证
你查文档         你/AI生成        你操作           你在 Godot 验证
"什么规格？"  →  "生成/获取"  →  "放对位置"   →  "显示正常？"
```

---

### B-1：确认规范

**目标**：确认新素材必须满足的技术规格。

**操作**：查阅 `docs/gdd.md` 对应章节。

| 素材类型 | 查阅章节 | 关键参数 |
|----------|----------|----------|
| 3D 模型 | 美术风格 → 角色设计 / 环境设计 | 文件格式、风格、比例、朝向 |
| 贴图 / 材质 | 美术风格 → 调色板 / 环境设计 / UI 风格 | 分辨率、色彩、用途 |
| UI 资源 | 美术风格 → UI 风格 | 字体、图标、面板风格 |
| 背景音乐 | 音频 → 音频技术参数 | 格式(.ogg)、采样率、响度 |
| 音效 | 音频 → 音频技术参数 | 格式(.wav)、时长、响度 |
| 数据表 | tdd.md → 数据管理 | 格式(JSON/CSV)、字段定义 |

**质量门**：

```
□ 知道目标格式和尺寸
□ 知道命名规则（snake_case）
□ 知道目标存放路径
```

---

### B-2：准备素材

**目标**：生成或获取符合规格的素材。

| 素材类型 | 工具 | 注意事项 |
|----------|------|----------|
| 3D 模型 | Blender / 外部 DCC / 现成资产 | 导出 `.glb` / `.gltf`，确认比例和朝向 |
| 贴图 / 材质 | 图像工具 / 材质工具 / AI 概念辅助 | 保持命名、分辨率与用途一致 |
| UI 资源 | 图像工具 / AI 概念辅助 | 仅用于 CanvasLayer/UI，不要混入 3D 世界资源 |
| 音乐 | Suno / Udio | 使用 gdd.md 中的 AI 音乐基底 prompt |
| 音效 | Suno / 专用工具 | 裁剪至合适时长，去除首尾静音 |
| 数据表 | Kimi / 手动 | 字段和格式严格按 tdd.md 定义 |

**产出指标**：

| 指标 | 要求 |
|------|------|
| 格式 | 与 B-1 确认的规格一致 |
| 命名 | snake_case，描述性命名（如 player.glb, player_albedo.png） |
| 风格 | 与项目已有素材一致（3D 风格、调色、材质方向等） |

---

### B-3：导入项目

**目标**：把素材放到正确位置，配置正确的导入设置。

**目录规范**：

```
assets/
├── models/
│   ├── characters/
│   ├── enemies/
│   ├── props/
│   └── environment/
├── textures/
│   ├── characters/
│   ├── environment/
│   └── ui/
├── materials/
├── audio/
│   ├── music/        ← .ogg 文件
│   └── sfx/          ← .wav 文件
├── fonts/            ← 字体文件
└── data/             ← JSON/CSV 数据表
```

**导入设置检查**：

| 素材类型 | Godot 导入设置 |
|----------|----------------|
| 3D 模型(.glb/.gltf) | 确认比例、朝向、动画导入与材质槽位 |
| 世界贴图 | 按实际用途确认压缩、色彩空间、Mipmaps |
| UI 贴图 | 按 UI 风格确认 Filter / 尺寸 / 清晰度 |
| 音乐(.ogg) | Loop = On（如需循环） |
| 音效(.wav) | Loop = Off |

**质量门**：

```
□ 文件在正确的 assets/ 子目录
□ 文件名 snake_case

## MVP 阶段简化规则

MVP 阶段的目标是快速验证核心玩法，以下规则允许你在 MVP 期间走捷径：

### 可以简化的步骤

| 步骤 | 正常流程 | MVP 简化 |
|------|----------|----------|
| 路径 A 的 A-6（文档同步） | 每个功能做完立刻更新 | 可以攒到 Sprint 结束批量更新 |
| 路径 C 的 C-2（全量 Review） | 审查所有代码 | 只 review P0 代码 |
| 路径 C 的 C-5（发布记录） | 详细 changelog | 简要记录即可 |
| git 分支 | feature branch（扩容后） | 直接在 main 开发 |

### 不能跳过的步骤

| 步骤 | 原因 |
|------|------|
| 路径 A 的 A-1（需求定义） | 不想清楚就写代码 = 返工 |
| 路径 A 的 A-2（设计收束） | 没有规格就没法拆任务 |
| 路径 A 的 A-5（集成验证） | F5 跑不起来一切白搭 |
| 路径 D 的 D-3（验证） | 没验证的 fix 不算 fix |

### MVP 进度追踪

建议在 `docs/tasks.md` 顶部维护一个 MVP 总览：

```markdown
## MVP 进度总览
目标：核心玩法可试玩
状态：in_progress

| 系统 | 来源 | 状态 | 阻塞项 |
|------|------|------|--------|
| 玩家移动 | gdd.md 核心机制 | merged | |
| 生命值 | gdd.md 核心机制 | in_progress | |
| 敌人 AI | gdd.md 核心机制 | draft | |
| 关卡加载 | tdd.md 核心系统 | draft | |
```

---

## 扩容阶段补充规则

当项目超过 MVP 进入 Alpha/Beta 时，以下规则生效：

### Git 分支策略

```
main              ← 稳定版本，随时可运行
  └── feat/xxx    ← 功能分支，从 main 创建
  └── fix/xxx     ← 修复分支，从 main 创建
```

- 新功能在 `feat/功能名` 分支开发
- 路径 C（并入主线）变为：feat 分支 → review → merge 到 main
- bug 修复在 `fix/描述` 分支，修完 merge 回 main

### EventBus 事件分类

当 EventBus 事件超过 8 个时，按域分组：

```gdscript
# event_bus.gd

# -- 玩家相关 --
signal player_damaged(amount: int)
signal player_died()

# -- 敌人相关 --
signal enemy_defeated(enemy_name: String)

# -- UI 相关 --
signal ui_notification_requested(text: String)

# -- 关卡相关 --
signal level_completed()
```
