> 本手册是 Track A（系统与代码开发）的操作指南。
> 按步骤执行即可完成从"想法"到"可运行代码"的完整流程。
>
> 系列手册：
> - **Track A: 系统/代码** ← 你在这里
> - Track B: UI/UX（Phase 0a 后可预研，手册待编写）
> - Track C: 视觉美术（C1 风格预研可在 0a 后开始，手册待编写）
> - Track D: 音频（需 Phase 0b，手册待编写）

---

> **MVP 简化**：MVP 阶段推荐双模型工作流 — ChatGPT（发散讨论）+ Claude Code（收束、实现、review，直接读项目文件）。
> Kimi 为可选，适合后期批量模板和配置生成。

```

gdd.md           tdd.md             tasks.md           source/          git commit
```

## A1：设计讨论

### 什么时候用

- 你要开始做一个新功能或新系统
- 你有一个模糊的想法，但还没想清楚"具体怎么玩"
- 你需要在几种方案之间做选择

### 什么时候跳过

- 功能非常明确，不需要讨论（如"加一个暂停菜单"）→ 直接去 A2
- 只是修 bug → 直接去 A4

### 操作步骤

**第 1 步：准备上下文**

从项目文件中复制以下内容，稍后粘贴到 prompt 里：

1. 打开 `docs/gdd.md`，复制 **"概览"** 章节全文
2. 打开 `docs/tdd.md`，复制 **"核心系统"** 章节中已有系统的名称列表（只要名称，不需要全部细节）

> **提示**：建议一次性复制所需上下文，避免多次切换文件。
> 如果使用 Claude Code，它可以直接读取项目文件，无需手动复制。

**第 2 步：打开 ChatGPT，粘贴以下 prompt**

```
## 上下文
[在这里粘贴 gdd.md "概览"章节]

## 已有系统
[在这里粘贴 tdd.md "核心系统"的系统名称列表]

## 任务
我想为这个游戏设计一个新系统：{{在这里填系统名称}}
玩家体验目标：{{在这里填一句话，描述玩家应该感受到什么}}

请：
1. 分析这个系统的核心交互是什么（玩家输入 → 系统响应 → 反馈）
2. 提出 2-3 种实现思路，列出各自优劣
3. 考虑和已有系统的关系（是否有耦合、冲突）
4. 推荐最适合独立开发者的方案，说明理由

## 输出格式
### 核心交互
- 玩家输入：...
- 系统响应：...
- 反馈循环：...

### 方案比较
| 方案 | 描述 | 优势 | 劣势 | 复杂度 |
|------|------|------|------|--------|

### 推荐方案
- 方案名：...
- 理由：...
- 与已有系统的关系：...
```

> **{{槽位}}说明**：所有 `{{...}}` 标记的地方是你需要替换的内容。删掉双花括号和里面的提示文字，填入你的实际内容。

**第 3 步：与 ChatGPT 迭代**

ChatGPT 的第一轮输出通常不是最终方案。追问以下问题来收敛：

- "方案 X 的边界情况有哪些？"
- "如果玩家做了 Y 会怎样？"
- "这个方案对后续扩展（如多人、存档）有影响吗？"
- "能把复杂度再降低吗？MVP 只需要哪些部分？"

**第 4 步：选定方案并保存**

- 从 ChatGPT 的输出中选定一个方案
- 把功能描述更新到 `docs/gdd.md` 对应章节
- 保留 ChatGPT 的推荐方案输出，下一步需要用

### 质量门（进入 A2 前必须满足）

```
□ 能用一句话说清楚这个功能是什么
□ 核心交互已明确：玩家做什么 → 系统怎么响应 → 什么反馈
□ 选定了一个方案，知道为什么选它
□ gdd.md 已更新
```

### 常见问题

| 问题 | 解决方案 |
|------|----------|
| ChatGPT 方案太复杂 | 明确告诉它"我是独立开发者，一个人做，给最简方案" |
| 讨论不收敛，越聊越散 | 停下来，自己先写一句话定义，再让 ChatGPT 评估可行性 |
| 不确定该不该做这个功能 | 问 ChatGPT："如果 MVP 只能有 3 个功能，这个应该排第几？" |

## A2：架构收束

### 什么时候用

- A1 选定方案后，需要变成工程可执行的规格
- 或者功能很明确不需要讨论（跳过了 A1），直接从这里开始

### 操作步骤

**第 1 步：准备上下文**

从项目文件中复制：

1. `docs/tdd.md` — **"架构概览"** 章节
2. `ai/context/architecture.md` — **全文**
3. `ai/context/coding_conventions.md` — **"命名规则"** 和 **"信号使用原则"** 两个章节

**第 2 步：打开 Claude（网页版或 Claude Code），粘贴以下 prompt**

> **工具选择**：
> - Claude Code：直接读取项目文件，无需复制上下文。推荐。
> - Claude web：需要手动复制下方上下文块。

```
## 上下文
[在这里粘贴 tdd.md "架构概览"章节]
[在这里粘贴 architecture.md 全文]

## 编码规范
[在这里粘贴 coding_conventions.md 的"命名规则"和"信号使用原则"]

## 任务
以下是我和 ChatGPT 讨论确定的功能方案：

功能名称：{{功能名称}}
选定方案：{{粘贴 A1 的推荐方案输出}}
补充说明：{{你的额外要求或约束，没有就写"无"}}

请把这个方案收束为工程规格。

## 输出格式
### 系统概述
- 名称：
- 一句话职责：
- 所属目录：source/features/xxx/

### 场景 & 节点树
```
[节点树结构]
```

### 信号设计
| 信号名 | 参数 | 触发时机 | 谁监听 |
|--------|------|----------|--------|

### 数据结构
| 数据 | 类型 | 说明 |
|------|------|------|

### 与已有系统的集成
- EventBus 事件：[列出需要新增的全局信号]
- 依赖的系统：[列出]
- 被依赖：[列出]

### 需要修改的现有文件
| 文件 | 修改内容 |
|------|----------|
```

**第 3 步：审查 Claude 的输出**

检查以下关键点：

- 节点树结构是否符合 `architecture.md` 的场景树组织方式
- 信号命名是否用了 past_tense（如 `health_changed` 而非 `change_health`）
- 是否有循环依赖
- EventBus 事件是否必要（参考通信决策树：广播型一对多 → EventBus，命令式一对一 → 直接引用，父子 → 信号向上/方法向下）

**第 4 步：保存规格**

- 把输出中的 **"系统概述"** 部分添加到 `docs/tdd.md` 的 **"核心系统"** 章节
- 保留完整输出，A3 需要用

### 质量门（进入 A3 前必须满足）

```
□ 系统有明确的一句话职责
□ 节点树结构确定，能直接在 Godot 里搭
□ 每个信号都有明确的触发时机和监听者
□ 与已有系统的集成方式明确（EventBus / 直接调用）
□ 没有循环依赖
□ tdd.md 已更新
```

### 常见问题

| 问题 | 解决方案 |
|------|----------|
| Claude 给的节点树太深 | 要求"节点树不超过 3 层，扁平优先" |
| 不确定该用 EventBus 还是直接信号 | 参考通信决策树（见下方） |
| 规格太粗，拆不出任务 | 要求 Claude "把每个节点的职责再展开，列出它需要的方法" |

> 通信方式选择（详见 ai/context/coding_conventions.md）：
> - 广播型事件（一对多）→ EventBus
> - 命令式调用（一对一，目标明确）→ 直接引用
> - 父子节点 → 信号向上，方法向下

## A3：任务拆解

### 什么时候用

- A2 工程规格完成后
- 需要把一个大功能拆成可执行的每日任务

### 操作步骤

**第 1 步：打开 Claude Code 终端**

Claude Code 可以直接读取项目文件，不需要手动复制上下文。

**第 2 步：输入以下 prompt**

```
请阅读 docs/gdd.md 中关于 {{功能名称}} 的描述，
结合 docs/tdd.md 的架构设计和 CLAUDE.md 的编码规范。

以下是这个功能的工程规格：
{{粘贴 A2 的完整输出}}

请把这个功能拆成开发任务，写入 docs/tasks.md。

## 拆分规则
- 每个任务 1-3 小时可完成
- 标明涉及的文件路径
- 标明任务间依赖关系
- 按 P0/P1/P2 优先级排列
- P0 = 核心逻辑，必须先做
- P1 = 完整功能需要，但不阻塞测试
- P2 = 打磨优化，可延后

## 输出格式（写入 docs/tasks.md）
## Sprint: {{功能名称}}
目标：{{一句话}}
截止：{{日期}}

### P0 — 核心
- [ ] 任务描述 | 文件: source/xxx/ | 依赖: 无
- [ ] 任务描述 | 文件: source/xxx/ | 依赖: 任务1

### P1 — 完整
- [ ] 任务描述 | 文件: source/xxx/

### P2 — 打磨
- [ ] 任务描述
```

> **提示**：如果 A2 的规格已经存放在某个文件中，可以直接让 Claude Code 读取，不需要粘贴。例如："请阅读 docs/tdd.md 中关于 XXX 的系统规格"。

**第 3 步：审查任务清单**

检查以下关键点：

- 每个任务是否足够具体（"实现 XXX" 不够具体，"创建 player.gd 并实现 move() 方法" 才够）
- P0 任务完成后，是否能在 Godot 里看到基本效果
- 任务之间的依赖是否正确（不能出现互相依赖）

**第 4 步：确认并保存**

- 确认 `docs/tasks.md` 已被 Claude Code 更新
- 如果需要调整，直接告诉 Claude Code 修改

### 质量门（进入 A4 前必须满足）

```
□ 每个任务有具体的完成描述
□ 每个任务标明了涉及的文件路径
□ P0 任务完成后能看到基本效果
□ 没有互相依赖的任务
□ tasks.md 已更新
```

### 常见问题

| 问题 | 解决方案 |
|------|----------|
| 任务拆得太细，10 几个子任务 | 告诉 Claude Code "合并相关的小任务，目标是 5-8 个任务" |
| 任务拆得太粗，一个任务要一天 | 告诉 Claude Code "这个任务太大，请拆成 2-3 个子步骤" |
| 不确定哪些是 P0 | 问自己："只做这些 P0，我能在 Godot 里跑起来看到效果吗？" |

## A4：代码实现

### 什么时候用

- tasks.md 中有待完成的任务
- 这是你日常开发中花时间最多的阶段

### 核心工作流

```
从 tasks.md 取一个任务
    │
    ├── 核心逻辑 / 跨文件修改 / 复杂 bug → Claude Code
    ├── 配置表 / 数值表 / CSV / JSON    → Claude Code（或可选 Kimi）
    └── 摆节点 / 调参数 / 试手感        → 你自己在 Godot 里做
    │
完成后立刻 F5/F6 验证
    │
通过 → 标记任务完成，取下一个
失败 → 修 bug（继续用 Claude Code）
```

### 操作步骤（Claude Code）

**第 1 步：从 tasks.md 选取一个任务**

按顺序取 P0 任务。确认没有未完成的依赖。

**第 2 步：在 Claude Code 终端输入以下 prompt**

```
请阅读以下文件获取上下文：
- CLAUDE.md（项目规则）
- ai/context/coding_conventions.md（编码规范）
- ai/context/architecture.md（架构约定）

当前任务（来自 docs/tasks.md）：
{{粘贴任务描述，包括文件路径和依赖信息}}

功能规格参考：
{{粘贴 A2 输出中与此任务相关的部分}}

## 要求
1. 严格遵循 coding_conventions.md 的脚本结构顺序和命名规则
2. 通信方式遵循决策树：广播型（一对多）→ EventBus，命令式（一对一）→ 直接引用，父子节点 → 信号向上/方法向下
3. @export 变量用于编辑器可调参数
4. 函数参数和返回值必须标注类型
5. 完成后告诉我需要在 Godot 编辑器里手动做什么（如连接信号、摆节点）
```

**第 3 步：审查 Claude Code 的输出**

快速检查：

- 脚本结构顺序是否正确（class_name → extends → signal → enum → const → @export → @onready → var → 生命周期 → 公共方法 → 私有方法 → 信号回调）
- 命名规则是否正确（snake_case 变量，PascalCase 类名）
- 是否有 Claude Code 提示你需要手动操作的部分

**第 4 步：在 Godot 编辑器中操作**

- 按 Claude Code 的提示在编辑器中摆节点、连信号
- F5 或 F6 运行，验证基本功能

**第 5 步：标记任务完成**

在 `docs/tasks.md` 中把 `- [ ]` 改为 `- [x]`。

### 操作步骤（Kimi — 批量生成，可选）

> 核心逻辑用 Claude Code 实现。批量模板、配置文件等重复性工作可选用 Kimi。

当任务涉及批量数据生成时（如物品表、掉落表、敌人属性表）：

**Prompt**：
```
请根据以下规则生成 {{内容类型}}：

格式要求：{{JSON / CSV / GDScript Dictionary}}
字段说明：
{{列出每个字段的名称、类型、取值范围、说明}}

数量：{{N}} 条
风格参考：
{{给一个完整的示例条目}}

## 要求
- 直接输出结果，不需要解释
- 确保数据合理，没有重复或矛盾
- 数值在指定范围内均匀分布
```

> **重要**：Kimi 的输出必须人工抽查至少 3 条数据的合理性。不要盲目信任批量结果。

### 质量门（每个任务完成后）

```
□ F5/F6 运行无报错
□ 任务描述中的功能可正常使用
□ 没有引入新的报错或警告
□ tasks.md 中该任务已标记完成
```

### 常见问题

| 问题 | 解决方案 |
|------|----------|
| Claude Code 生成的代码报错 | 把错误信息直接粘贴给它："运行后报以下错误，请修复：[错误信息]" |
| 不确定该自己做还是让 AI 做 | 规则：逻辑写代码的活给 AI，视觉/手感调参的活自己做 |
| Kimi 生成的数据质量差 | 给更详细的示例（至少 2-3 条完整示例），明确说明取值范围 |
| 一个任务做不下去，卡住了 | 先跳过，做下一个无依赖的任务，之后再回来 |

## A5：Code Review

### 使用模式

Track A 支持两种使用模式：

- **独立模式**：直接从 A1 开始，A5 完成 review + git commit。
- **迭代模式**：被迭代手册 A-4 调用进入 A4/A5。此模式下 A5 只做 review 和 Godot 验证，**跳过 git commit**（提交统一在迭代手册 Path C 处理）。

### 什么时候用

- 每次准备 git commit 之前（独立模式）
- 一个完整功能的所有任务都完成后做一次全量 review

### 操作步骤

**第 1 步：在 Claude Code 终端输入以下 prompt**

```
请 review 当前的代码改动（git diff）。

## 检查维度
1. **功能正确性**：逻辑是否实现了预期功能
2. **规范一致性**：是否符合 ai/context/coding_conventions.md
3. **架构合规性**：是否符合 ai/context/architecture.md 的通信规则
4. **安全性**：是否有空引用、类型错误、资源泄漏
5. **性能**：是否有明显的性能问题（如 _process 中做重计算）

## 输出格式
### 问题列表
| 级别 | 文件:行号 | 问题描述 | 修复建议 |
|------|-----------|----------|----------|
| 🔴 必须修 | | | |
| 🟡 建议改 | | | |
| 🟢 可选优化 | | | |

### 提交前检查清单
- [ ] 场景能正常加载，无报错
- [ ] 新功能核心路径可走通
- [ ] 没有破坏其他系统
- [ ] 信号连接正确
- [ ] @export 变量在编辑器里能用
- [ ] 符合编码规范
- [ ] 没有硬编码魔法数字
```

**第 2 步：处理 Review 结果**

| 问题级别 | 行动 |
|----------|------|
| 🔴 必须修 | 立刻修复，修完后重新 review |
| 🟡 建议改 | 本次修复或记入 tasks.md 下个 sprint |
| 🟢 可选优化 | 记入 tasks.md 作为 P2 |

**第 3 步：提交前最终检查**

在 Godot 编辑器中完整走一遍：

```
□ F5 运行主场景 → 无报错
□ 新功能的核心路径可走通（从头到尾操作一遍）
□ 切换到其他已有功能 → 没有被破坏
□ 检查编辑器中的 @export 变量 → 能正常调整
```

**第 4 步：Git 提交**（独立模式执行，迭代模式跳过）

> 如果是通过迭代手册 A-4 调用的，跳过此步骤，提交统一在 Path C 处理。

所有检查通过后：

```bash
git add [具体文件]
git commit -m "feat: 实现 XXX 系统的 YYY 功能"
```

### 提交信息规范

```
feat: 新功能
fix: 修复 bug
refactor: 重构（不改变功能）
chore: 杂项（配置、文档更新）
```

### 质量门（提交前必须满足）

```
□ 所有 🔴 问题已修复
□ Godot 编辑器中 F5 无报错
□ 核心功能路径可走通
□ git diff 中没有调试代码（print 语句、临时变量）
```

## 文档更新时机

> **MVP 文档规则**：开发期间只维护 tasks.md + tdd.md 的增量变化。
> 完整文档同步推迟到迭代手册路径 C（并入主线）时执行。

采用两点更新模型：

| 时机 | 触发条件 | 更新内容 |
|------|----------|----------|
| **设计冻结**（A2 完成后） | 方案确定，准备拆任务 | 一次性更新 `gdd.md` / `tdd.md` / `architecture.md` |
| **实现偏差**（代码完成后） | 实现与设计有出入 | 仅更新偏离部分的文档 |

其他文件更新参考：

| 文件 | 什么时候更新 |
|------|-------------|
| `docs/tasks.md` | A3-A4 持续更新 |
| `docs/changelog.md` | 发版时 |
| `CLAUDE.md` | 项目结构变化时 |

> 如果更新了 gdd.md 或 tdd.md 的核心章节，记得执行 **M1 项目变更协议**
> （详见 docs/dev_workflow.md "七、Meta — 项目变更协议"章节）。
